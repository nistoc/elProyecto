# Требования к API-сервисам (backend)

Для всех API-проектов экосистемы (первый и последующие сервисы) действуют следующие архитектурные требования и лучшие практики.

---

## 1. Чистая архитектура (Clean Architecture, Роберт Мартин)

- **Слои и правило зависимостей:** зависимости направлены внутрь. Внешние слои зависят от внутренних; внутренние слои не знают о внешних.
- **Рекомендуемые слои (от внутреннего к внешнему):**
  - **Domain:** сущности, доменные типы, без зависимостей от фреймворков и инфраструктуры.
  - **Application (Use Cases):** интерфейсы репозиториев и сервисов, сценарии применения (use cases), DTO для входа/выхода. Зависит только от Domain.
  - **Infrastructure:** реализации интерфейсов (репозитории, работа с файлами, БД, внешние API). Зависит от Application (интерфейсы) и Domain.
  - **Web / Presentation (API):** контроллеры, модели запросов/ответов для HTTP, маппинг в DTO. Зависит от Application и при необходимости от Domain; не содержит бизнес-логики.
- **Итог:** бизнес-логика и правила живут в Domain и Application; инфраструктурные детали (файл, PostgreSQL, Ninject и т.д.) — во внешних слоях.

---

## 2. Вертикальное слайсинг по функциональности

- **Организация по фичам:** код и логика каждой фичи лежат **внутри одной папки** в проекте. Одна фича — одна папка (слайс).
- **Структура каталогов (обязательная):**
  - **`Features/<FeatureName>/`** — одна папка на фичу (например, `Features/Ideas/`, `Features/Auth/`).
  - **Внутри слайса** лежит всё, что относится к фиче: контроллер(ы), интерфейсы (Application), реализации (Infrastructure), модели/DTO этой фичи. Допускаются подпапки внутри слайса (например, `Application/`, `Infrastructure/`, `Models/`) для порядка, но допустимо и плоское размещение файлов в одном каталоге слайса.
  - **Корневых папок** типа `Controllers/`, `Application/`, `Infrastructure/`, `Models/` с кодом из разных фич — **не использовать**: это горизонтальное деление по слоям, а не вертикальные слайсы.
- **Цель:** при добавлении или изменении фичи все правки локализованы в одной папке; слайсы не зависят друг от друга по коду; упрощение навигации и ревью.
- **Composition Root** (Program.cs, Ninject-модули) остаётся в корне и ссылается на типы из слайсов только для регистрации привязок.

---

## 3. SOLID и интерфейсы

- **SRP (Single Responsibility):** один класс — одна причина для изменения. Контроллеры только принимают запрос и вызывают use case; логика хранения — в репозитории; валидация — в отдельном компоненте или use case.
- **OCP (Open/Closed):** расширение через новые классы и интерфейсы, без изменения существующего кода (например, новая реализация репозитория для PostgreSQL).
- **LSP (Liskov Substitution):** реализации интерфейсов подставляемы друг вместо друга без нарушения контракта.
- **ISP (Interface Segregation):** узкие интерфейсы под клиента (например, `IIdeasReadRepository` и `IIdeasWriteRepository` при необходимости, либо один `IIdeasRepository` с чётким контрактом).
- **DIP (Dependency Inversion):** зависимость от абстракций (интерфейсов), а не от конкретных классов. Слой Application определяет интерфейсы (например, `IIdeasStore`); Infrastructure предоставляет реализации; в контроллерах и use cases инжектируются только интерфейсы.
- **Обязательно:** для каждого репозитория или внешнего сервиса — отдельный интерфейс в Application (или Domain при чисто доменных контрактах) и конкретная реализация в Infrastructure. Контроллеры и use cases зависят только от интерфейсов.

---

## 4. DI-контейнер: Ninject

- **Использовать Ninject** в качестве контейнера зависимостей (вместо встроенного Microsoft.Extensions.DependencyInjection или вместе с ним через адаптер).
- **Composition Root:** все привязки интерфейсов к реализациям задаются в одном месте (модуль Ninject или метод конфигурации при старте приложения).
- **Регистрация:** интерфейсы привязываются к конкретным классам; при необходимости задаётся время жизни (Singleton, Transient, Scoped в терминах ASP.NET Core — через соответствующие биндинги Ninject).
- **Ссылка:** пакет `Ninject.Web.AspNetCore` для интеграции с ASP.NET Core; для .NET 10 использовать совместимую версию.

---

## 5. Контроллеры в API

- **Использовать контроллеры** (наследование от `ControllerBase`, атрибуты маршрутизации и HTTP-методов), а не только минимальный API (MapGet/MapPost в одном файле).
- **Один контроллер на ресурс или на вертикальный слайс:** например, `IdeasController` для `/api/ideas`, при появлении авторизации — `AuthController` для `/api/auth/*`.
- **Ответственность контроллера:** приём HTTP-запроса, опционально валидация входной модели, вызов use case или сервиса (через интерфейс), маппинг результата в HTTP-ответ (JSON, коды состояния). Бизнес-логика в контроллере не размещается.
- **Соглашения:** маршруты в стиле REST (`GET /api/ideas`, `GET /api/ideas/{id}`, `POST /api/ideas`); возврат типизированных моделей и при ошибках — единообразный формат (например, ProblemDetails).

---

## 6. Источник пакетов NuGet

- Все пакеты для .NET-проектов загружать **только из официального источника nuget.org**: `https://api.nuget.org/v3/index.json`.
- В корне репозитория (или решения) использовать `nuget.config` с явным указанием этого источника, при необходимости с `<clear />`, чтобы не подтягивать сторонние feed'ы.

## 7. Дополнительные практики

- **Валидация входных данных:** использовать FluentValidation или DataAnnotations для моделей запросов; возвращать 400 с перечнем ошибок при невалидных данных.
- **Обработка ошибок:** единый подход (exception filter, middleware или поведение в use case); для API — предпочтительно ProblemDetails (RFC 7807); логирование исключений через `ILogger`.
- **Асинхронность:** все операции I/O — асинхронные (`async`/`await`); передавать `CancellationToken` из контроллера в use case и репозиторий.
- **Логирование:** внедрять `ILogger<T>` в сервисы и use cases; не логировать чувствительные данные (пароли, токены).
- **Конфигурация:** строки подключения, пути к файлам и т.п. — из `IConfiguration` (или Options), не захардкожены в коде.
- **Тестируемость:** возможность подменять все зависимости (репозитории, внешние сервисы) через интерфейсы для unit- и интеграционных тестов.

---

## Итоговый чек-лист для нового или рефакторинга API

1. Слои Clean Architecture: Domain, Application (интерфейсы + use cases), Infrastructure (реализации), Web (контроллеры).
2. Вертикальные слайсы: структура по фичам (Ideas, Auth, …).
3. Интерфейсы для всех репозиториев/внешних сервисов; зависимости только от абстракций (SOLID, DIP).
4. Ninject настроен как DI-контейнер; привязки в composition root.
5. API реализован через контроллеры (ControllerBase); один контроллер на ресурс/слайс.
7. Валидация, единый формат ошибок, async + CancellationToken, логирование, конфигурация через IConfiguration.
8. Пакеты — только из nuget.org (см. п. 6).

При создании или доработке API ориентироваться на этот документ; эталонная реализация при необходимости задаётся в плане или в каталоге API первого сервиса в репозитории.
Репозиторий может содержать один или несколько сервисов: основной сервис в репозитории определеять по отдельному документу README.md, который должен создаваться и наполняться при внесении любых изменений проект API. Если сервис внутри репозитория создается один - то все изменения описываем в changelog
