# RENTGEN — Виртуальная абстрактная модель (требования к реализации)

## Принципы

- **Независимость фичи от ядра сервиса:** отдельный слайс / read-модель; фича не вызывает pipeline и не зависит от деталей транскрипции.
- **Разделение чтения и записи:** виртуальная модель строится как read-модель над хранилищем сущностей; запись выполняют только pipeline и SubmitJob.
- **Единый запросный слой:** один слой «виртуальная модель» над хранилищем сущностей.
- **Доступ по идентификатору и по семантическим ключам:** поддержка доступа по id сущности и по тегам/меткам (семантический ключ).
- **Динамический статус по каждой сущности независимо от других ключей:** для UI с периодическим обновлением — возможность получать статус одной работы по jobId и списки с полным статусом по каждой записи.

## Требования

- Рассчитано на опрос с частотой **0,01–10 Hz**.
- Кеширование неизменных частей: разный TTL для активных (Running/Pending) и завершённых (Completed/Failed) состояний.
- Инвалидация кеша при изменении сущностей.
- Опционально: rate limit для защиты от перегрузки (например макс. 10 req/s на клиента, 429 при превышении).

## Дизайн

### Read-модель и индексы

- Виртуальная модель = read-модель + индексы по семантическим ключам (теги, метки).
- Интерфейс запросов (например `IJobQueryService` или аналог для других сущностей):
  - получение по id;
  - список с фильтрами (status, время, тег);
  - запрос по семантическому ключу с пагинацией.
- Ответы содержат **полный статус по каждой сущности** в списке.

### Модель узлов (нод)

- Виртуальная модель может быть представлена **деревом узлов**.
- Обновление при начале/окончании работы: `StartNode` / `CompleteNode` → `startedAt`, `completedAt`, `status`.
- Иерархия по `parentId`, агностичная к глубине; контур `scopeId`.
- Контракт узла: `id`, `parentId`, `scopeId`, `kind`, `status`, `startedAt`, `completedAt`, `updatedAt`, `children`.

### Реализация через атрибуты XRay

- Атрибуты XRay помечают **методы бизнес-логики**, которые фигурируют или обновляют виртуальную модель (какой метод соответствует шагу в дереве узлов). В коде Agent04 используются `[XRayNode(Operation)]` (Ensure, Start, Complete) на методах пайплайна, выполняющих `EnsureNode` / `StartNode` / `CompleteNode`.
- Опционально: перехватчик по атрибуту и контексту может выполнять обновление узлов вместо явных вызовов в бизнес-коде; иерархия выводится из контекста (текущий nodeId = parent для новой ноды). Атрибуты **не влияют** на порядок вызовов и зависимости.
- **Важно:** теги `Activity` (System.Diagnostics) для трассировки (OpenTelemetry и т.п.) **не являются** XRay в смысле виртуальной модели; XRay в Agent04 — именно атрибуты на методах.
- В API виртуальной модели параметр **tag** означает идентификатор или название **узла** (node id), по которому запрашивают статус: GET /jobs/{id}/nodes?tag=nodeId возвращает только этот узел. Фильтр списка заданий по семантическому ключу — отдельный параметр **semanticKey** (GET /jobs/query?semanticKey=...).

## Архитектура

- Отдельный вертикальный слайс (например `Features/JobQuery`), зависимость только от интерфейса чтения хранилища.
- Слой кеширования в Infrastructure (IMemoryCache или аналог, только nuget.org).
- REST и gRPC endpoint'ы для запросов.
- При расширении — возможность выноса read-слоя в отдельный процесс/реплику без изменения контракта.
- Запись в модель узлов — только в границах «начало/конец работы» (EnsureNode, StartNode, CompleteNode), без привязки к глубине дерева.
