## 1. Что реализовано полностью

- **Этапы 0–10:** остов, Domain, Audio, Cache, OpenAI client, Output, Merger, Pipeline, REST API (POST/GET jobs, список, Location, 202), gRPC (SubmitJob, GetJobStatus, StreamJobStatus), IJobStatusStore, мониторинг, теги в REST и gRPC, кеширование (CachingJobStatusStore), Ninject (Agent04Module), модель узлов (INodeModel, INodeQuery, GET /jobs/{id}/nodes), RENTGEN_IMPLEMENTATION.md и скрипт регистрации факта, README, CHANGELOG. (CLI в текущей реализации присутствует; запланировано **удаление** поддержки CLI — см. раздел 4.)
- **REST:** ProblemDetails для 400/404, OpenAPI с тегами и описаниями для виртуальной модели (RENTGEN).
- **Виртуальная модель:** запрос по jobId, список по тегу (GET /jobs/query), дерево узлов по scopeId, кеш с TTL и инвалидацией.

## 3. OpenAPI и описание работы с RENTGEN

- В OpenAPI добавлены **теги** для группировки операций:
  - **Jobs** — создание и мониторинг заданий (POST/GET jobs, список).
  - **Virtual model (RENTGEN)** — виртуальная модель: запрос по тегу и дерево узлов.
- Для тега **Virtual model (RENTGEN)** в документе задано **описание**: назначение (запросы по семантическому ключу, иерархия узлов по job), типичное использование (GET /jobs/query, GET /jobs/{id}/nodes), частота опроса, ссылка на docs/RENTGEN_IMPLEMENTATION.md.
- У операций добавлены **summary** и **remarks**:
  - GET /jobs/{id}/nodes — иерархия шагов (job → chunking → transcribe → chunk-0..N → merge), параметр tree, поля узла (id, parentId, scopeId, kind, status, startedAt, completedAt, updatedAt), ссылка на RENTGEN_IMPLEMENTATION.md.
  - GET /jobs/query — запрос по семантическому ключу (тег), фильтры, полный статус по каждой записи, пригодность для опроса 0.01–10 Hz.

Внешний сервис по OpenAPI может: увидеть группу «Virtual model (RENTGEN)», прочитать описание тега и операций, понять, как получать список заданий по тегу и как получать дерево узлов по job id (flat или tree).

## 4. Рабочий каталог (песочница) и абсолютные пути

### 4.1. Назначение

- Перевести Agent04 на работу с **единым корнем — workspace_root** (абсолютный путь в конфиге). Все пути к конфигам, аудио и результатам (JSON/MD) задаются **относительно workspace_root** и разрешаются агентом внутри этого дерева.
- Обеспечить **общий каталог-песочницу**, доступный нескольким агентам: один кладёт исходные файлы, Agent04 читает аудио и пишет транскрипт/JSON в тот же каталог, другой агент читает результат. Файл в Agent04 не передаётся — агент обращается только по путям внутри workspace_root.
- Пример: `workspace_root` = `C:\trash\spanish\elProyecto\agent-browser\runtime`; проекты и файлы лежат в подкаталогах (например `project1/config.json`, `project1/audio.mp3`). Позже планируется переход на S3; пока — локальные папки.

### 4.2. Контракт (требования к реализации)

- **Workspace root (песочница):** один **обязательный** базовый каталог на инстанс Agent04. Задаётся **только в конфиге** (поле `workspace_root` — абсолютный путь). Разные инстансы могут иметь разные `workspace_root`. В API (REST/gRPC) поле `workspace_root` **не передаётся**.
- **Старт приложения:** при запуске Agent04 проверяет существование каталога `workspace_root`; если каталог не найден или недоступен — приложение завершается с ошибкой (без приёма запросов).
- **configPath:** путь к файлу конфига задания. Разрешается **относительно workspace_root** (все файлы проекта — конфиги, аудио, транскрипты — располагаются внутри дерева текущего обрабатываемого проекта под workspace_root).
- **Входной файл (inputFilePath):** в API (REST/gRPC) и в конфиге — **только относительный путь** относительно workspace_root. Абсолютные пути в `inputFilePath` **не допускаются** (допустимый единственный абсолютный путь в контракте — сам `workspace_root` в конфиге).
- **Выходные артефакты:** пути к транскрипту и JSON разрешаются относительно workspace_root (или относительно каталога проекта внутри workspace). Для **сохранения истории** предыдущих запусков имена файлов транскрипта должны быть **уникальными** (например `{base}_{jobId}_transcript.md` или с временной меткой); остальные файлы (промежуточные, кеш, один и тот же `openai_response.json` и т.п.) допускается перезаписывать.
- **Вспомогательные каталоги** (cache_dir, split_workdir и т.д.): задаются относительными путями к workspace root (или к каталогу проекта). Кеш/чанки располагаются под workspace root.
- **Создание каталогов:** workspace root к моменту запуска приложения **уже должен существовать**. Подкаталоги (для выходов, кеша, чанков) **создаёт Agent04**, если их не хватает.
- **Поведение:** Agent04 не принимает аудио в теле запроса; агент читает и пишет файлы только внутри дерева workspace_root по путям, разрешённым из конфига и API.

### 4.3. Сценарий с несколькими агентами

- Workspace root задаётся в конфиге инстанса Agent04 (например `C:\...\agent-browser\runtime`). Агент 1 записывает в подкаталоги этого workspace аудиофайлы и конфиги.
- Оркестратор или агент 2 вызывает Agent04: `POST /api/transcription/jobs` с `configPath` и `inputFilePath` — **относительными** путями к workspace_root (например `project1/config.json`, `project1/audio.mp3`). Или передаётся идентификатор/имя проекта внутри workspace_root, для которого запускается транскрипция или приходят управляющие сигналы.
- Agent04 разрешает пути относительно `workspace_root` из конфига, читает аудио и конфиг, пишет транскрипт и JSON в тот же workspace (имена выходных файлов — по правилам контракта, с обеспечением уникальности для истории).
- Агент 3 читает транскрипт и JSON по путям из статуса задания (`md_output_path`, `json_output_path`).

### 4.4. Перспектива S3

- Контракт «workspace root в конфиге + все пути относительно него» в дальнейшем позволит подменить файловый слой на S3 (workspace root = bucket/prefix), без изменения семантики для вызывающих.

### 4.5. Связь с существующими пунктами плана

- В разделе 2.1 (REST API) зафиксирован приоритет: не multipart для загрузки файла, а поддержка **workspace_root в конфиге и путей относительно него**; multipart при необходимости — опциональное расширение позже.
- Доработка «песочница и относительные пути» (настоящий раздел 4) входит в перечень планируемых правок наравне с остальными пунктами раздела 2.

### 4.6. План реализации (для разработки)

- **Конфиг:** добавить **обязательное** поле `workspace_root` (абсолютный путь) в JSON-конфиг (или в глобальные настройки — appsettings/переменная окружения — один раз на инстанс). Использовать как единственную базу для разрешения всех относительных путей (`configPath`, `inputFilePath`, `md_output_path`, `raw_json_output_path`, `cache_dir`, `split_workdir` и др.).
- **Старт приложения:** при запуске проверять существование каталога `workspace_root`; если каталог не найден — завершать приложение с ошибкой (без приёма запросов).
- **API (REST/gRPC):** в теле SubmitJob **не** передавать `workspace_root` (он только в конфиге). Передавать `configPath` и `inputFilePath` как **относительные пути к workspace_root**. Контракт REST и gRPC одинаковый: в proto `SubmitJobRequest` поле `workspace_root` не добавляется; в комментарии к proto указать, что пути в запросе интерпретируются относительно `workspace_root` из конфига инстанса. В TranscriptionGrpcService применять ту же логику разрешения путей, что и в REST (чтение workspace_root из конфига/настроек).
- **Разрешение configPath:** при приёме запроса разрешать `configPath` относительно `workspace_root` (полный путь к файлу конфига = `Path.Combine(workspace_root, configPath)` или аналог). Конфиг задания загружается из этого пути.
- **Разрешение inputFilePath:** разрешать только относительно workspace_root; если в запросе передан абсолютный путь — возвращать 400 (Bad Request). В конфиге `file`/`files` — только относительные к workspace_root.
- **Pipeline:** в TranscriptionPipeline при формировании путей (mdPath, jsonPath, manifestPath, workdir для чанков, wav, cache) всегда использовать workspace_root как базу для относительных путей из конфига.
- **Уникальность имён транскрипта:** формировать имена выходных файлов транскрипта так, чтобы сохранять историю (например `{base}_{jobId}_transcript.md` или с временной меткой); остальные артефакты допускается перезаписывать.
- **Создание подкаталогов:** при записи выходов и вспомогательных файлов создавать подкаталоги под workspace_root, если их не существует.
- **Удаление CLI:** убрать поддержку запуска в режиме CLI (аргумент `--config`, ветка «если CLI — RunCliAsync и выход»). Agent04 запускается **только как веб-сервис** (HTTP + gRPC); все вызовы транскрипции — через REST или gRPC. Подробности — в п. 4.7 («Что будет удалено»).
- **Документация:** обновить README и пример конфига: обязательный `workspace_root`, все пути относительные, пример многоАгентного сценария; убрать описание запуска через CLI.

### 4.7. Принятые решения (по уточнениям перед реализацией)

- **Имя поля:** везде используется только `workspace_root` (в конфиге); в API поля с таким именем нет.
- **configPath:** разрешается **относительно workspace_root**; все файлы проекта (конфиги, аудио, транскрипты) располагаются внутри дерева обрабатываемого проекта под workspace_root.
- **Источник workspace_root:** задаётся **только в конфиге**; из API не передаётся. Это позволяет запускать несколько инстансов Agent04 с разными workspace_root. Из API приходят только идентификатор/путь проекта внутри workspace_root и управляющие сигналы.
- **Отсутствующий workspace root:** приложение проверяет существование каталога при старте и **завершается с ошибкой**, если каталог не найден.
- **Валидация путей:** `inputFilePath` допускается только как **относительный** путь к workspace_root. Абсолютные пути в `inputFilePath` **не принимаются**; единственный допустимый абсолютный путь в контракте — значение `workspace_root` в конфиге.
- **gRPC:** контракт совпадает с REST: в proto **не** добавляется поле `workspace_root` в SubmitJobRequest; в proto указывается в комментарии, что пути в запросе интерпретируются относительно workspace_root из конфига. TranscriptionGrpcService использует ту же логику разрешения путей, что и REST.
- **Уникальность выходных файлов:** обеспечить уникальные имена для файлов транскрипта (чтобы сохранять историю предыдущих `transcript.md`); остальные файлы допускается перезаписывать.
- **Создание каталогов:** workspace root к моменту запуска уже должен существовать; подкаталоги (для выходов и т.п.) **создаёт Agent04**, если их нет.
- **Ссылки на план миграции:** в подразделах 2.1–2.5 номера разделов приведены в виде «раздел N плана миграции», чтобы не путать с разделами данного документа.

#### Что будет удалено или изменено при реализации (требование «не поддерживаем отсутствие workspace_root»)

- **Удалить:** разрешение `inputFilePath` относительно **каталога файла конфига** (`Path.GetDirectoryName(Path.GetFullPath(configPath))`). Сейчас в REST и gRPC относительный input комбинируется с директорией configPath — эта логика убирается.
- **Удалить:** ветку «при отсутствии workspace_root» (fallback на текущую логику). Поддержки запуска без `workspace_root` не будет: конфиг обязан содержать `workspace_root`, при старте проверяется его наличие и существование каталога.
- **Изменить:** разрешение `configPath` — больше не от текущей рабочей директории и не «как есть»; только относительно `workspace_root` (полный путь = workspace_root + configPath). Проверка существования файла конфига — по этому полному пути.
- **Удалить поддержку CLI:** убрать парсинг аргументов `--config` и флаг режима CLI; убрать ветку при старте «если CLI — вызвать RunCliAsync и выйти»; удалить метод `RunCliAsync` и логику разрешения configPath только для CLI. После изменений приложение при запуске всегда поднимает веб-сервер (HTTP + gRPC) и не предлагает одноразового консольного запуска по конфигу.

## 5. Итог

- Основная функциональность плана реализована: остов, фичи транскрипции, REST, gRPC, мониторинг, теги, кеш, Ninject, модель узлов, RENTGEN-документ и скрипт, документация.
- Не реализованы или опциональны: встроенный конфиг и multipart в POST, X-Caller-Id, webhook/callback, gRPC QueryJobs, исходящий gRPC, отдельный слайс JobQuery, rate limit, TotalChunks/ProcessedChunks в контракте, XRay.
- Запланирована доработка **«песочница (workspace_root) и относительные пути»** (раздел 4): обязательный workspace_root в конфиге, все пути относительно него, переиспользование общего каталога между агентами, подготовка контракта к переходу на S3.
- Запланировано **удаление поддержки CLI**: Agent04 — только веб-сервис (HTTP + gRPC); одноразовый запуск по `--config` не поддерживается; вызовы — через REST или gRPC.
- Работа с API виртуальной модели (RENTGEN) описана в OpenAPI (теги, описания операций и тега), чтобы внешние сервисы могли интуитивно понять, как получать данные по этой логике.
